# 16 — 生活记录三件套：小票、月历、纪念日卡

## 这是什么

三个把「过日子」搬进系统的小工具，共同点是：**数据存服务端一份，人和 AI 从各自的入口读写同一份**。

- **今日小票**：超市小票风格的每日清单，人自己记，AI 聊天时也能帮记（帮记的条目带标记）
- **小月历**：月经周期记录，服务端算统计（平均周期/预测下次/延迟提前），三端口径一致
- **纪念日卡**：纪念日当天打开前端，弹一张 AI 提前亲笔写好的小卡片，一年一张，历年可翻

单独看都是十几行 SQL + 一个卡片组件的小活，但踩的坑有共性，值得记。

## 共同架构

每个工具 = 一张表 + 一组 REST 路由 + 前端卡片 + AI 工具（前端工具/MCP 各一份）。统计类的计算**一定放服务端**：小月历的 computeStats 在路由文件里实现并导出，MCP 复用同一个函数——否则三个入口各算各的，「预计下次」能给出三个日期。

```js
// routes/period.js
function computeStats(logs) { /* 平均周期、预测、delta_days */ }
module.exports = router;
module.exports.computeStats = computeStats;  // mcp.js 复用
```

统计还要做离群过滤：周期 >120 天的间隔（漏记了几个月）、经期 >15 天的时长（忘了记结束）不进平均。

## 坑一：允许补记历史，就别用「最新一条」

小月历上线当天就翻车：用户乱序补记一年的历史（先记了最近一次，再回头补四月的），四月那条永远显示「进行中」关不上。

根因是三处代码都写了 `logs[0] && !logs[0].end_date`——「进行中的 = 最新一条」这个假设在乱序录入下直接崩掉。修法：

```js
// ✗ const ongoing = logs[0] && !logs[0].end_date ? logs[0] : null
// ✓ const ongoing = logs.find((l) => !l.end_date) || null
```

配套改动：POST 守卫只拦「进行中的那次之后又开新的一次」，补记更早的历史日期放行；列表里每条「进行中」的行都带自己的操作按钮，别让唯一的按钮只指向头部。

**通用教训：凡是允许补记历史的功能，查找目标一律用谓词 find，不能取头部。**

## 坑二：等宽字体的全宽字符

小票用 `▮▯` 拼装饰条码，结果整张小票横向溢出——这两个是全宽字符，在等宽字体里占两倍宽度，把「纸」撑破了。装饰性字符先在目标字体里量宽度再用。

## 纪念日卡：机制比卡片本身有意思

```sql
CREATE TABLE anniversary_cards (
  anniversary_id INTEGER NOT NULL,
  year INTEGER NOT NULL,     -- 周年卡存年份；里程碑卡复用这列存天数（520 不会和年份撞车）
  message TEXT NOT NULL,
  UNIQUE(anniversary_id, year)
);
```

- `GET /anniversaries/today`：按用户时区的 MM-DD 匹配周年；不中再算里程碑天数（第 100/520/1000/1314 天，起始日=第 1 天）
- 前端挂载时查一次，命中且当天没看过（localStorage 记日期）才弹；拉取失败静默，绝不影响聊天
- 卡片内容是 AI 在 CC 里提前 `curl POST` 写好的——**一年只写当年这张**，不预写未来。这既是实现细节也是态度：系统不替未来许诺

## 复盘：这类工具的验收清单

1. 统计口径三端一致吗？（算在服务端、被所有入口复用）
2. 乱序补记会卡死吗？（find 不取头部、守卫放行历史日期）
3. AI 帮记的条目有署名吗？（added_by 字段 + 前端小标记，谁记的一目了然）
4. 拉不到数据时会不会拖垮主功能？（静默降级）
